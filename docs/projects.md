# 项目理念与需求

## 项目目标

构建一个面向 MyBatis / MyBatis-Plus 的数据库字段加密增强插件，在尽量不改造业务 Mapper 和 Service 代码的前提下，实现敏感字段的透明加密、条件查询改写和结果自动解密。

这个项目不是“通用数据库加密平台”，而是一个聚焦字段级数据保护的 MyBatis 扩展能力，强调：

- 低侵入：业务层尽量继续按明文字段编程
- 可扩展：算法、规则声明方式、查询辅助列都可以扩展
- 可落地：优先支持最常见的写入、等值查询、LIKE 查询场景
- 可控风险：对不安全、不可靠、语义不明确的 SQL 行为主动失败
- 面向国内场景：默认能力优先兼容国密算法与商密建设要求

## 核心理念

### 1. 业务代码无感，但规则必须显式

插件应尽量让业务开发者继续使用原有实体字段和 Mapper 方法，不要求手工加密解密。

但“无感”不代表“隐式猜测”。对任何加密字段、辅助查询字段、LIKE 查询字段，都必须通过注解或配置显式声明，避免误改写 SQL。

### 2. 先保证正确，再追求覆盖面

如果某种 SQL 场景无法保证改写后的查询语义仍然正确，那么应直接报错，而不是放行。

宁可少支持，也不要制造隐蔽的数据错误。

### 3. 存储加密与查询能力解耦

主字段负责落密文，查询能力通过辅助列或 LIKE 列提供。不要试图用同一列同时兼顾：

- 安全存储
- 等值检索
- 模糊检索
- 排序
- 范围比较

这几类能力在设计上应明确拆分。

### 3.1 默认算法优先采用国密

默认主加密算法应优先采用 `SM4`，默认辅助等值查询算法应优先采用 `SM3`。

原因：

- 更贴近国内商用密码和等级保护建设要求
- 便于在政企项目中直接落地
- 降低业务侧二次定制默认算法的成本

同时保留非国密算法作为兼容选项，而不是默认选项。

### 4. 配置体验向 ShardingSphere 靠拢，但保持实现轻量

规则配置应尽量具备清晰的层次结构，便于理解、维护和批量管理；同时保留注解声明方式，降低单表接入成本。

### 5. 安全日志优先于调试便利

涉及敏感字段的 SQL 输出、参数打印、调试日志，默认应输出脱敏值、哈希值或占位标记，不能直接输出明文和真实密文。

### 6. 安全优先，但不能无视性能

默认算法选择不能只考虑“是否安全”，还要考虑在常见单字段加密场景中的吞吐与延迟成本。

当前阶段的性能原则：

- 主加密采用对称算法，避免默认使用非对称算法做字段加密
- 等值检索采用单向摘要列，避免查询时对密文做反复解密
- 算法实例和 Provider 初始化应尽量复用，降低重复创建开销
- 先保证正确和可控，再逐步做压测和热点优化

## 需要解决的问题

当前项目要解决的不是“是否加密”，而是下面这些工程问题：

- 敏感字段如何在数据库中以密文形式保存
- 业务代码如何继续以明文字段读写对象
- 等值查询如何在不暴露明文的情况下可用
- 模糊查询如何通过独立的辅助列实现
- MyBatis / MyBatis-Plus 自动生成 SQL 如何被统一拦截和改写
- 查询结果如何在返回业务层前自动解密
- 当 SQL 场景不受支持时，如何明确失败并给出可理解的错误

## 功能要求

### 必须具备

- 支持 Spring Boot 自动装配
- 支持 MyBatis / MyBatis-Plus 接入
- 支持通过注解声明加密规则
- 支持通过配置文件声明加密规则
- 支持主加密列写入
- 支持辅助等值查询列
- 支持 LIKE 查询辅助列
- 支持查询结果自动解密
- 支持算法 SPI 扩展
- 支持 SQL 参数脱敏日志
- 默认使用 `SM4 + SM3`
- 保留 `AES + SHA-256` 兼容能力
- 自定义算法只需实现接口并注入 Spring Bean
- 支持按开关自动扫描带 `@EncryptField` 的实体注解
- `@EncryptTable` 不是强制项，未声明时优先兼容 MyBatis-Plus `@TableName`
- 支持“同表存储”和“独立加密表存储”两种字段策略

### 需要优先覆盖的 SQL 类型

- `INSERT`
- `UPDATE`
- `DELETE`
- `SELECT`

### 需要优先覆盖的查询行为

- 等值查询
- `IN` 查询
- `LIKE` 查询

## 非目标

以下能力不应作为当前阶段的默认承诺：

- 任意复杂 SQL 的自动正确重写
- 所有数据库方言的兼容
- 分布式数据库或分库分表协调能力
- 完整替代 ShardingSphere 加密模块
- 加密字段上的排序与范围检索
- 完整接管所有第三方 SQL 日志组件

## 风险控制要求

### 1. 不允许静默降级

遇到已声明加密字段但操作不受支持时，必须直接失败，不能静默忽略。

### 2. 不允许猜测字段含义

没有规则的字段不做自动推断，不根据字段名猜测是否需要加密。

### 3. 不允许输出敏感明文

日志、异常、调试输出中都要避免泄露敏感数据。

### 4. 不允许伪支持

对 `ORDER BY`、范围查询、复杂表达式等语义不可靠场景，不提供“有限支持”式模糊行为，应明确标记为不支持。

## 当前阶段实现要求

当前仓库的代码实现应至少满足以下工程标准：

- 核心能力集中在可测试的独立组件中，而不是散落在拦截器里
- SQL 改写逻辑、规则注册逻辑、结果解密逻辑彼此解耦
- 自动配置与核心逻辑分层清晰
- 默认算法可直接开箱使用，但允许用户替换
- 必须有最小可运行测试覆盖主链路
- README 必须提供 Maven / Gradle / 多模块三类引入示例
- README 必须明确“默认国密”与“不能直接等同合规完成”这两个事实
- 所有 SPI 接口必须有类级和方法级注释
- 所有核心类必须有职责说明，至少覆盖拦截器、规则注册中心、SQL 改写器、结果解密器、自动配置类
- 复杂逻辑必须注释设计意图、适用边界和失败策略，避免维护时只能从实现反推语义

## 当前模块结构

当前仓库已经调整为多模块结构，职责划分如下：

- `common`：核心模块，承载算法 SPI、注解模型、配置模型、元数据注册、SQL 改写、结果解密与 MyBatis 拦截器能力
- `bom`：BOM 模块，统一管理对外发布的 starter 版本
- `spring-starter/spring3-starter`：Spring Boot 3.x 主接入模块，当前文档和测试都以该模块为主
- `spring-starter/spring2-starter`：Spring Boot 2.x 兼容接入模块
- `spring-starter`：starter 聚合父模块，用于组织 Spring 相关子模块

对外使用时的推荐顺序：

- Spring Boot 3 项目优先依赖 `mybatis-like-sharephere-support-spring3-starter`
- Spring Boot 2 项目依赖 `mybatis-like-sharephere-support-spring2-starter`
- 多个 starter 需要统一版本时，通过 `mybatis-like-sharephere-support-bom` 做版本约束
