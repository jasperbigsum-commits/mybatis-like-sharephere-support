# 迁移生产上线操作手册

[中文](migration-production-runbook.zh-CN.md) | [English](migration-production-runbook.en.md)

## 这份文档适合什么时候看

当迁移方案、DDL、确认策略都已经准备完成，准备进入生产变更窗口时看这份文档。

如果你还处在方案设计阶段，建议先阅读：

1. [存量迁移指南](migration-guide.zh-CN.md)
2. [迁移游标设计指南](migration-cursor-design.zh-CN.md)
3. 然后再回到本文按 checklist 执行

## 使用方式

这份文档偏操作手册，不是概念说明。

推荐做法：

- 研发、DBA、运维在上线前共同过一遍
- 执行时按 checklist 逐项勾选
- 出现异常时直接跳到“失败后处置规范”

## 角色分工

### 研发负责人

负责：

- 提供最终实体规则、游标定义、备份策略
- 生成并解释 DDL、风险清单、迁移命令
- 解释错误码和失败后的处置方案

上线前必须确认：

- 所有覆盖式字段都已经评估是否需要 `backupColumn(...)`
- `MigrationSchemaSqlGenerator` 输出已经过人工审核
- 灰度批次的 `batchSize`、`verifyAfterWrite`、checkpoint 目录已经固定
- 已明确是否允许断点续跑、谁可以删除或保留状态文件

### DBA

负责：

- 执行或审核 DDL
- 检查表结构、索引、锁风险、执行窗口
- 在失败后协助核查主表与独立表数据一致性

上线前必须确认：

- 新增列、独立表、长度扩容 SQL 已全部执行完成
- 相关索引、唯一约束、外表主键策略与设计一致
- 批量更新不会明显冲击主业务高峰
- 已准备回查 SQL，用于核对迁移前后记录

### 运维或发布负责人

负责：

- 控制发布时间窗口
- 保证迁移任务单实例执行
- 保存日志、checkpoint、确认文件和执行报告

上线前必须确认：

- 任务命令、环境变量、数据源指向都正确
- 同一任务不存在并发实例
- checkpoint 目录可持久化且不会被临时清理
- 失败时谁负责暂停、谁负责恢复已经明确

## 执行前 checklist

以下内容建议逐项勾选：

- 目标环境是正确的数据源，不是测试库或影子库
- 本次迁移涉及的实体、表、字段范围已经冻结
- DDL 已执行并核对
- 风险确认文件已生成并审核
- checkpoint 目录为空，或旧状态文件已确认可复用
- 所有覆盖式字段都已确认明文仍可恢复
- 批量大小已采用灰度值，不是首次就上全量大批次
- `verifyAfterWrite` 已开启
- 应急联系人、回滚数据来源、值班群都已就位

## 首次执行 checklist

推荐顺序：

1. 先执行小批量灰度，不要直接跑全量
2. 观察日志中的读批、更新、校验、提交是否连续正常
3. 核对 `MigrationReport` 的 `migratedRows`、`skippedRows`、`verifiedRows`
4. 抽样检查主表与独立表结果
5. 确认没有出现异常错误码后，再逐步放量

灰度阶段至少抽查：

- 同表模式下 `storageColumn`、`assistedQueryColumn`、`likeQueryColumn`
- 独立表模式下主表引用列与外表 hash/cipher/like
- 覆盖式字段的备份列是否已成功写入
- checkpoint 是否推进到最新已提交游标

## 执行中暂停条件

出现以下任一情况，建议立即暂停，不要继续放量：

- 出现 `PLAINTEXT_UNRECOVERABLE`
- 出现连续的 `VERIFICATION_VALUE_MISMATCH`
- 独立表开始出现重复引用、孤儿记录或主外表不一致
- checkpoint 长时间不推进，但任务仍在重试
- 业务侧出现锁等待明显升高、慢 SQL 激增、主链路告警

暂停后不要立刻做的事：

- 不要先删 checkpoint 文件
- 不要先重启多个实例试图“撞过去”
- 不要把已经改写过的源列继续当明文手工修补

## 失败后处置规范

### 通用恢复原则

迁移中断后优先选择“修复问题后重跑同一任务”，不要先删除 checkpoint。

继续执行的前提：

- 使用同一个数据源
- 使用同一个实体/表和游标列
- 使用同一个 checkpoint 目录
- 使用同一个字段范围和备份策略

如果任务已经有已提交批次，下一次会从已提交 checkpoint 后继续。
如果失败发生在事务内，未提交批次会回滚，下次会重新处理。

一键批量迁移同样适用这个规则。`executeAllRegisteredTables()` 会按表分别保存状态，二次执行时已完成且目标态完整的表不会重复改写；目标态不完整的表会按幂等规则补偿。

### 遇到 `CHECKPOINT_LOCKED`

说明：

- 同一迁移任务已有其他实例在执行

处理：

1. 先确认是否确实已有实例在跑
2. 保留当前状态文件，不做人工修改
3. 关闭重复实例，只保留一个执行入口

### 遇到 `VERIFICATION_VALUE_MISMATCH`

说明：

- 当前批次写后校验未通过，目标态和预期派生值不一致

处理：

1. 暂停任务
2. 对失败记录做主表、外表、备份列抽样核对
3. 检查算法配置、字段映射、业务写入干扰
4. 修复后再从现有 checkpoint 继续

### 遇到 `PLAINTEXT_UNRECOVERABLE`

说明：

- 源列已经被部分迁移改写，但当前又无法从备份恢复原始明文

处理：

1. 立即停止重跑
2. 从备份、历史表、审计记录恢复原始明文
3. 修复残缺的派生列或独立表记录
4. 补充备份列策略后再继续

这是强停止类错误，不建议靠“多跑几次”解决。

### 从备份字段补偿恢复

如果覆盖式字段配置了 `backupColumn(...)`，迁移器会自动优先从备份列读取原始明文。

适用场景：

- 源列已经被 hash、like、cipher 或独立表引用值覆盖
- 备份列仍然保存原始明文
- 密文列、like/hash 列或独立表记录缺失，需要继续补偿

操作方式：

1. 确认备份列里是原始明文
2. 保持迁移定义中的 `backupColumn(...)` 不变
3. 直接重跑同一个迁移任务

不需要调用单独的恢复 API。

## 上线完成后复核 checklist

迁移完成后建议至少完成以下复核：

- `MigrationReport.status` 为 `COMPLETED`
- `verifiedRows` 与预期一致
- 关键字段抽样查询通过
- 接口联调确认未出现明文/密文错位
- checkpoint、确认文件、执行日志已归档
- 失败批次和人工修复记录已留痕

## 明确禁止的操作

以下做法最容易导致二次事故：

- 未审核 DDL 就直接在生产执行迁移任务
- 未配置备份就对覆盖式字段反复重跑
- 人工修改 checkpoint 游标试图跳过异常数据
- 多实例并发执行同一迁移任务
- 把 `verifyAfterWrite(true)` 当成 dry-run
- 看到异常后先删状态文件再分析原因
